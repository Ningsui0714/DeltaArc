## 结果

本次修复了上一轮评价里最明确的交付阻断项：依赖安全告警已清零，并新增 GitHub Actions 质量门禁。当前 `npm audit --audit-level=moderate` 返回 `found 0 vulnerabilities`。

## 改动

1. 依赖安全修复

- 执行 `npm audit fix`，将 Express 4 链路中的 `path-to-regexp` 修到 `0.1.13`，将 `postcss` 修到 `8.5.12`。
- 将 Vite 从 5.x 升到 7.x，将 React Vite 插件升到 5.x，避开 `vite <=6.4.1` / `esbuild <=0.24.2` 的开发服务器漏洞范围。
- 当前依赖版本：`vite@7.3.2`、`@vitejs/plugin-react@5.2.0`、`express@4.22.1`、`path-to-regexp@0.1.13`、`postcss@8.5.12`、`esbuild@0.27.7`。

2. CI 门禁

新增 `.github/workflows/ci.yml`，在 push 到 `main/master` 或 PR 时执行：

```bash
npm ci
npm run typecheck
npm test
npm run build
npm run verify:fixtures
npm run harness:preflight
npm audit --audit-level=moderate
```

CI 使用 Node `24.x`，与本地验证环境 `v24.13.0` 对齐。

## 风险与回滚

- Vite 从 5.x 升到 7.x 属于大版本升级，但当前项目的 Vite 配置很薄，验证命令全部通过，短期风险可控。
- 若后续发现 Vite 7 兼容问题，可回滚 `package.json` / `package-lock.json` 中的 Vite 与 `@vitejs/plugin-react` 版本，再重新执行验证；但这会重新暴露 audit 风险。
- CI 新增后，未来依赖漏洞会更早失败，这是预期的质量门禁行为。
