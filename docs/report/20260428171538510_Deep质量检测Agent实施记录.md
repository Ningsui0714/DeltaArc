# Deep 质量检测 Agent 实施记录

时间：2026-04-28 17:15:38

## 结果

已把 Preflight 的 `deep` 模式改成两段式调用：第一段沿用原发布前推演，第二段新增“质量检测 Agent”复核结果质量、证据支撑、评论覆盖、发布风险和可执行性；`quick` 模式保持单次调用快路径。

## 改动

### 1. 后端编排

- `server/lib/preflight/runPreflightSimulation.ts`
  - `quick`：仍只调用一次 `provider.generateJson`。
  - `deep`：先生成基础推演，再用基础结果构造质量检测 prompt，第二次调用 provider。
  - 第二阶段只合并 `qualityCheck`、`publishSafetyReview` 和 `warnings`，不重写 `simulatedReplies`、`interventions`、`pushModel` 等主结果。
  - 如果第二阶段失败：保留第一阶段结果，标记 `degraded=true`，追加质量检测失败 warning，并要求人工复核。

### 2. 质量检测 Agent Prompt

- `server/lib/preflight/prompt.ts`
  - 新增 `buildPreflightQualityAgentPrompt(request, candidate)`。
  - 质量 Agent 只输出：
    - `qualityCheck`
    - `publishSafetyReview`
    - `warnings`
  - 检查点包含四类人群覆盖、视觉证据、0-100 分制、动作可执行性、旧案例串台、发布风险护栏。

### 3. Prompt 审查问题修复

已处理 `20260428170626787_豆包Prompt审查报告.md` 中的 P0/P1：

- 在 `resultJsonContract` 中补齐运行时仍使用的字段：
  - `contentRead`
  - `imageInsight`
  - `pushModel`
  - `confidence`
- 将 `simulatedReplies` 契约从 `tier/comment` 统一到运行时字段名：
  - `relevanceTier`
  - `text`
- 强化 `intervention.evidenceSource` 约束，要求引用 `visualRead / contentPromise / simulatedReplies / riskReview / publishSafetyReview` 中的具体来源。

新增/增强测试覆盖：

- prompt 契约包含 `contentRead / imageInsight / pushModel / confidence`。
- `quick` 仍只调用一次 provider。
- `deep` 会调用两次 provider，并把质量检测 Agent 的 `qualityCheck / publishSafetyReview / warnings` 合并到最终结果。
- 质量检测 Agent 失败时保留基础结果并标记 degraded。
- parser 能把顶层 warnings 镜像到 `qualityCheck.outputWarnings`。

- `qualityCheck.needsHumanReview` 能把发布闸门推到 `revise`。
- critical red flag 能把发布闸门推到 `hold`。

## 风险与回滚

- 风险：`deep` 模式耗时和成本上升。
  - 回滚：只需把 `deep` 的质量检测调用关闭，`quick` 不受影响。
- 风险：质量检测 Agent 误判。
  - 控制：第二阶段不重写主结果，只影响质量检查和发布护栏。
- 风险：prompt 变长。
  - 控制：本轮先修 schema 对齐；后续可再拆 quick/deep 契约减负。
