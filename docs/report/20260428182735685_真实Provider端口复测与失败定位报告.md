## 真实 Provider 验证

### 1. 最小多模态探针

命令直接调用豆包/火山方舟 `/responses`，使用同一张压缩后的 `xhs-genshin-return-guide.jpg`。

结果：

- `status=200`
- 耗时：约 `9142ms`
- 返回了图片场景 JSON

结论：API Key、网络、模型名、图片 base64 和多模态接口本身可用。

### 2. 页面完整链路第一次

URL：

`http://127.0.0.1:4100/?preflightCase=xhs-genshin-return-guide`

结果：

- 图片成功读取
- 表单填充成功
- 小红书平台和作者名正确
- 等待约 `151341ms`
- 进入兜底结果
- 页面提示：`豆包/火山方舟调用超时，已等待 150000ms。`

### 3. 优化后复测

已做优化：

- 将 2MB 级 PNG 压缩为 180-340KB JPG
- BrowserUse 样例改为加载 JPG
- quick 模式 prompt 改为紧凑草图契约
- quick 模式远端只需返回 4 条评论、1 个风险、1 个 intervention，服务端负责补齐覆盖结构

复测结果：

- 图片成功读取
- 表单填充成功
- 等待约 `72764ms`
- 进入兜底结果
- 页面提示：`fetch failed`
- console error：0

## 判断

真实 provider 不是完全不可用，因为最小多模态探针 9 秒成功；失败点在完整发布前试映接口调用上。当前最可能原因是：

- 完整 app prompt 仍然比最小探针重很多，远端响应不稳定。
- 火山方舟 `/responses` 在长输出 JSON 场景下偶发连接失败或超时。
- 当前服务端兜底能保护页面不崩，但不能算真实 provider 跑通。

## 当前可用状态

- `4100/4101` 服务已启动，可继续手动观察。
- mock provider 四个小红书样例已全部 Browser Use 通过。
- 真实 provider 最小多模态探针通过。
- 真实 provider 页面完整推演未通过，不能把这轮记为真实完成态。

## 我的感受

这次问题不像样例内容或端口问题，更像真实 provider 对“长 JSON + 图片 + 多模块输出”的稳定性不够。页面兜底做得还算稳，用户能看到结果和错误原因，但这不应该被包装成真实推演成功。要让它真正能演示，我会优先继续把真实 quick 链路拆成两步：先让豆包只做视觉读片和 4 条核心评论，再由服务端生成结构化增长、安全和质量模块。
