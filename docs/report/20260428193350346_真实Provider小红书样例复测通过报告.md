## 定位与修复

复测中先发现 `xhs-coop-game-night` 两类真实失败：

- 第一次失败：豆包返回 `OutputTextSensitiveContentDetected`，原因是输出审核拦截。
- 第二次失败：模型返回了未闭合 JSON，原因是 quick 合约仍过大。

修复后 quick 链路改为“真实模型短草图 + 服务端结构补齐”，真实 provider 的调用时间从此前 60-150 秒降到约 23-31 秒，稳定性明显提升。

## 我的感受

这轮问题的核心不是“真实 provider 能不能跑”，而是“真实 provider 不适合一次吐完整发布报告”。把远端职责缩成判断、评论、风险和动作草图，服务端负责结构化补齐，效果更稳，也更像一个产品化 agent：模型做高价值判断，程序做确定性收口。

当前样例已经更像小红书发布前测试：封面、账号名、评论区、发布安全和行动建议都能被页面吃进去。后续如果要继续提质，我会优先做 provider 原始输出的可观测日志和每个样例的视觉快照归档，这样排查会更快。
