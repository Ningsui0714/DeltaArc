# DeltaArc 项目体检评价报告

时间：2026-04-27 18:00:31

## 总体评价

DeltaArc 当前已经从早期“内容风洞 / 策略诊断”转成一条更聚焦的主链路：发布前社媒预演器 / KOC Growth Lab。项目不是空壳，已经具备前端可演示界面、Express 异步任务 API、共享类型契约、Doubao/火山方舟 provider、mock provider、fixture harness 和较完整的本地验证脚本。

综合判断：它现在是“可演示、可比赛提交候选、还不适合直接当多人生产服务”的状态。最强的地方是产品闭环和验证意识，最需要补的是任务基础设施、失败语义、输入边界和前端状态隔离。

## 亮点

- 产品定位清楚：主入口已收敛到 `PreflightStudioPage`，用户能在一个页面完成素材输入、手机预览、模拟评论和增长动作查看。
- 技术栈轻：React 18 + Vite + TypeScript + Express，没有过度引入框架，构建产物小。
- 契约意识强：`shared/preflightSimulation.ts` 承担前后端共享类型和结果归一化，mock provider 与 fixture harness 能保证稳定回归。
- 验证链路完整：`typecheck`、`test`、`build`、`verify:fixtures`、`harness:preflight` 均可跑通。
- 演示体验成立：本地生产服务用 `PREFLIGHT_PROVIDER=mock` 启动后，Browser Use 实测可以打开页面、载入示例、完成传播推演并展示结果。

## 主要风险

1. 远端模型失败会静默降级成成功结果。
   - 位置：`server/lib/preflight/runPreflightSimulation.ts`
   - 影响：豆包鉴权失败、超时或 JSON 解析失败后，任务仍进入 completed，用户可能把本地 mock/fallback 当成真实模型输出。
   - 建议：增加 `degraded` 状态或显式 `fallbackReason`，让 UI 直接展示“远端失败，已用本地模拟兜底”。

2. 任务入口缺少队列、限流和取消机制。
   - 位置：`server/routes/preflightSimulations.ts`
   - 影响：重复点击或脚本请求会直接放大成外部模型调用、内存占用和成本风险。
   - 建议：抽出统一 job runner，至少加全局并发、workspace 限流、容量上限和取消/忽略旧任务机制。

3. Job store 只在内存中保存完整请求。
   - 位置：`server/lib/preflight/preflightJobStore.ts`
   - 影响：base64 图片会在进程内保留最多 30 分钟；进程重启任务状态全丢；没有容量上限。
   - 建议：短期限制图片和 job 数量；中期把 request/result 分层存储，避免把大 payload 长时间挂在 Map 中。

4. Preflight 请求 schema 偏宽松。
   - 位置：`shared/preflightSimulation.ts`
   - 影响：workspaceId、文本长度、图片大小、mime、外部 URL 没有足够校验；错误请求会被默认值悄悄纠正。
   - 建议：增加 workspaceId 正则、文本长度上限、dataUrl/base64 大小限制、mime allowlist 和 URL 策略。

5. 前端轮询可能被旧请求回写。
   - 位置：`src/hooks/usePreflightSimulation.ts`
   - 影响：用户在任务运行时点击“清空结果”或“载入示例”，旧的 in-flight polling 仍可能回写 job/result。
   - 建议：加 `activeRunIdRef` 或 `AbortController`，所有状态写入前校验当前 run。

6. 示例数据和真实结果展示边界不够清楚。
   - 位置：`src/pages/PreflightStudioPage.tsx`
   - 影响：未生成结果时也显示示例评论、示例互动指标和涨粉动作，评委或用户可能误以为已经完成真实推演。
   - 建议：无结果时明确标记“示例预览”，或改成空态/示例卡，不直接显示预测数值。

7. 历史 sandbox API 无条件暴露。
   - 位置：`server/index.ts`
   - 影响：旧分析、baseline、impact scan 和 workspace 写操作扩大部署暴露面。
   - 建议：用 feature flag 收口旧路由，或拆成 legacy server/API。

## 验证记录

- `npm run typecheck`：通过。
- `npm test`：通过，114 passed / 0 failed。
- `npm run verify:fixtures`：通过，输出 `Fixture verification passed.`。
- `npm run harness:preflight`：通过，输出 `Verified campus-coffee-demo`。
- `npm run build`：通过，前端与服务端构建完成。
- Browser Use：通过，`http://127.0.0.1:5001/` 可打开；点击“一键载入示例”与“开始传播推演”后出现“传播发展推演已完成”“传播人群分层”“涨粉动作”“模拟评论区”。

## 结论

- 演示：可以演示，建议使用 `PREFLIGHT_PROVIDER=mock` 或提前确认豆包 key/endpoint。
- 比赛提交：可以作为提交候选，但提交前最好先修复轮询回写、示例/真实结果混淆、远端 fallback 语义。
- 生产化：还缺队列限流、持久化/容量控制、严格输入校验、旧 API 暴露收口和更清晰的 degraded 状态。

## 优先改进顺序

1. 修 `usePreflightSimulation` 旧轮询回写，避免演示现场出现旧结果反弹。
2. UI 标清示例数据和真实推演结果，降低误读。
3. 调整 preflight fallback 语义，远端失败显式 degraded。
4. 给 preflight API 增加请求校验、图片大小限制和外部 URL 策略。
5. 增加 job 并发/容量限制，并把旧 sandbox API 放到 feature flag 后。
