# 豆包 Prompt 审查报告

时间：2026-04-28 17:06:26

## 结论

当前豆包 prompt 的主方向合理：角色定位清楚，能区分 KOC 增长、校园提交、品牌发布三类场景；约束了 JSON-only、简体中文、0-100 分制、固定人群覆盖、8 条模拟评论和发布安全审查。现有 provider 与解析兜底链路也能通过相关单测。

但 prompt 还不是“稳定高质量”的状态，主要问题是 JSON 契约和运行时 schema 不完全一致，容易让豆包漏掉部分旧字段，只能由解析层用通用兜底补齐，导致结果看似完整但信息密度下降。

## 关键发现

1. `server/lib/preflight/prompt.ts` 的 `resultJsonContract` 只要求输出 V2 新字段，没有列出运行时 `PreflightSimulationResult` 仍保留的 `contentRead`、`imageInsight`、`pushModel`、`confidence`。
   - 影响：系统提示要求“视觉弱时写到 imageInsight 和 confidence.limitations”，但用户 prompt 的 JSON 契约没有这些字段，豆包大概率不会稳定输出。
   - 当前兜底：`shared/preflightSimulation.ts` 会从 `visualRead`、`audienceDistribution` 推导部分旧字段。
   - 风险：前端展示会完整，但 `imageInsight`、`confidence.limitations` 容易变成通用模板，不够贴合真实图片和内容。

2. `simulatedReplies` 契约使用 `tier` 和 `comment`，运行时类型使用 `relevanceTier` 和 `text`，解析层虽然兼容两种写法，但 prompt 没有说明别名关系。
   - 影响：这不是阻断问题，但会增加模型输出漂移和测试样例维护成本。
   - 建议：统一要求输出运行时字段名，或在契约中明确 `tier -> relevanceTier`、`comment -> text` 的兼容关系。

3. prompt 对“数量”和“安全审查”的约束比较强，这是优点，但 quick 模式依然要求完整大 JSON，`userPrompt` 约 9000 字符，真实 provider 已有耗时偏长记录。
   - 影响：复杂案例、多图或 deep 模式可能增加超时、截断、JSON 修复概率。
   - 建议：quick 模式保留核心字段，非首屏字段可由服务端派生；deep 模式再展开完整 schema。

4. prompt 对“不能承诺真实流量、不能刷量、发布安全”的边界做得不错，且已补 0-100 分制约束；这部分是合理的，不建议删。

## 建议优先级

P0：补齐 prompt 契约，让 `resultJsonContract` 和 `PreflightSimulationResult` 对齐，至少加入 `contentRead`、`imageInsight`、`pushModel`、`confidence`，或明确这些字段由服务端生成、模型不要输出。

P1：统一字段命名，减少 `tier/comment` 与 `relevanceTier/text` 双轨。

P1：给 prompt 增加“证据引用”约束，例如每个 `intervention.evidenceSource` 必须来自 `visualRead / contentPromise / simulatedReplies / riskReview` 中的一项，降低空泛建议。

P2：拆 quick/deep 两套契约，quick 减少字段和文本长度，deep 保留全量输出。
