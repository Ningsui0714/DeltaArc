# API 多次调用架构评估报告

生成时间：2026-04-28 17:05:30

## 结论

当前架构里，“旧 sandbox 分析链路”已经是多阶段、多候选、多验证的 API 编排；“当前主产品 preflight simulation”本质上仍是一次豆包/方舟大 JSON 调用，外层用异步 job 展示多个进度阶段。

判断：多次调用可以做到比单次调用效果更好，但前提不是“多叫几次模型”，而是把调用拆成有明确质量职责的链路：先做证据 grounding，再生成差异化候选，再用 verifier / reverse check 做筛选，最后只让 finalizer 补齐前端结构。否则多次调用只会增加延迟、成本和信息损耗。

## 当前架构证据

### 1. 当前主产品链路：进度多阶段，模型单次调用

主入口在 `server/routes/preflightSimulations.ts`：

- `POST /api/preflight-simulations` 创建异步 job 后调用 `runPreflightSimulation`。
- `GET /api/preflight-simulations/:jobId` 轮询 job 状态。

核心执行在 `server/lib/preflight/runPreflightSimulation.ts`：

- 前面依次上报 `content_read`、`image_read`、`push_model`、`reply_simulation`、`synthesis` 等阶段。
- 真正远端模型调用集中在 `provider.generateJson(...)` 一次。
- 失败时落到本地 mock fallback，再经过 `normalizePreflightResult` 补齐结构。

Provider 在 `server/lib/preflight/doubaoProvider.ts`：

- 使用 `POST ${DOUBAO_BASE_URL}/responses`。
- 输入包含 system prompt、user prompt 和图片。
- 要求 `json_object` 输出。
- quick 输出上限约 7200 tokens，deep 输出上限约 9000 tokens。

这说明当前主体验的“多阶段”主要是产品进度表达，不是多个真实推理子任务。

### 2. 旧 sandbox 链路：已经有真实多调用编排

主编排在 `server/lib/orchestration/index.ts`：

```text
dossier -> specialists -> synthesis -> refine
```

已经包含：

- `runDossierStage`：grounding、多个 dossier candidate、dossier-select。
- `runSpecialistStages`：多个 specialist 按并发度执行。
- `runSynthesisStage`：future slice 与 action brief 并行，并在 reasoning 模式下做 reverse check。
- `runRefineStage`：可选最终审校。
- checkpoint / resume：阶段失败后保留已完成结果，支持续跑。

这条链路的方向是对的，但历史文档也指出风险：链路长、dossier 过早压缩、schema 太大时，后续阶段可能消费二手摘要并放大信息损耗。

## 是否多次调用更好

会更好，适合以下场景：

1. 输入含图片、长文本、品牌安全边界或比赛提交材料，需要先稳定抽取事实。
2. 输出要覆盖多目标：人群分发、评论模拟、增长动作、发布安全、质量检查。
3. 单次大 JSON 容易漏字段、串台、十分快/百分制混淆，或评论覆盖不足。
4. 结果需要可解释：哪些是输入证据，哪些是模型推断，哪些需要人工复核。

不会更好，甚至会变差的场景：

1. 只是把同一个 prompt 拆成多轮串行总结，没有 verifier。
2. 每个阶段都输出大而全 schema，导致成本和失败面扩大。
3. 后续阶段只吃前一阶段摘要，不回看原始输入和证据。
4. 没有候选选择、规则校验、降级标记和本地 fallback。

## 推荐架构

对当前主产品建议采用“双轨”：

### Quick：保留单次调用快路径

适用于演示、低成本、低风险输入。

```text
Input + Images -> Single Preflight JSON -> Normalize -> UI
```

保留原因：

- 改动小。
- 成本最低。
- 对短内容足够快。
- 现有 mock / harness / 页面展示都已围绕它稳定。

### Quality：新增多调用质检路径

适用于 deep 模式、图片较多、发布风险高、比赛提交、自定义品牌边界较多的输入。

```text
Input Normalize
  -> Evidence Pack / Visual Grounding
  -> Candidate Simulation x 3
  -> Verifier
  -> Finalizer
  -> PreflightSimulationResult
```

建议职责：

1. `Evidence Pack / Visual Grounding`
   - 只抽取事实、视觉元素、内容承诺、限制条件、缺失证据。
   - 不下最终结论。

2. `Candidate Simulation x 3`
   - `audience_candidate`：人群分发与评论覆盖。
   - `growth_candidate`：选题、标题、封面、评论互动和账号成长动作。
   - `safety_candidate`：发布风险护栏、红旗、必须修复项。

3. `Verifier`
   - 检查候选是否引用 evidence。
   - 检查弱相关 / 误推评论是否覆盖。
   - 检查是否有跨主题串台、过度承诺、分数尺度错误。
   - 输出 blocking issues 和通过/降级标记。

4. `Finalizer`
   - 只负责把通过验证的内容合并成现有 `PreflightSimulationResult`。
   - 不重新发明主要判断。

## 最小落地方案

不建议一次把主链路完全改成复杂 multi-agent。建议分三步：

1. 先把 `deep` 模式接入多调用，`quick` 仍保留单次 provider。
2. 先实现 `Evidence Pack -> Candidate x 3 -> Verifier -> Finalizer`，中间结构只放服务端内部，不改前端协议。
3. 用现有 fixture 和真实 provider 案例做 A/B：单次调用 vs 多调用，人工评分后再决定是否推广到 quick。

## 验收指标

建议用 10-20 个固定案例评估：

- 评论覆盖：`core / broad / weak / misfire` 每类至少 2 条。
- 串台率：宿舍、校园、游戏、品牌运营等场景之间不得泄漏旧主题词。
- 发布安全：高风险项必须进入 `publishSafetyReview.redFlags` 或 `mustFixBeforePublish`。
- 分数尺度：所有分数字段稳定为 0-100。
- 可执行性：每条 intervention 必须包含具体修改面和可复制示例。
- 延迟：quick 不超过当前单次链路；deep 可接受更慢，但需要在 UI 明确展示真实阶段。
- 成本：deep 多调用成本不超过单次 deep 的 2.5-3.5 倍，超过则需要减少 candidate 数量。

## 风险与回滚

- 风险：调用数增加导致耗时变长。
  回滚：保留 `quick` 单次调用，deep 多调用用配置开关控制。

- 风险：阶段间信息损耗。
  回滚：每个候选和 verifier 都携带原始输入摘要与 evidence pack，不只消费上一阶段自由文本。

- 风险：verifier 自己误判。
  回滚：verifier 只做评分、阻断项和修复建议，不单独拍板；关键硬约束继续用本地规则校验。

- 风险：前端协议被中间过程污染。
  回滚：中间结构全部服务端内部化，最终仍输出 `PreflightSimulationResult`。

## 架构决策建议

建议采用“单次调用快路径 + 多调用质量路径”的混合架构。

一句话：当前主产品如果只追求演示稳定，单次调用够用；如果追求更稳的真实输出质量，多次调用值得做，但必须按“证据、候选、验证、定稿”的质量系统来做，而不是简单堆 agent 角色。
