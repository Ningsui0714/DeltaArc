# DeltaArc 项目评价报告

时间：2026-04-28 14:32:15

## 总体结论

DeltaArc 当前已经不是简单原型，而是一套围绕“发布前试映场”的可演示产品：前端体验、后端异步 job、多模态 provider、mock 回归、schema 归一化和测试体系都已经成型。我的评价是 **7.5/10，可参赛演示，可继续产品化；但还不建议直接按生产服务上线**。

最核心的价值点是：它没有做成泛泛的聊天助手，而是把产品收敛到“内容发布前预判首批反馈、误推人群、风险与改稿动作”。这个定位清楚，适合校园 AI 产品或增长工具演示。

## 亮点

1. 主链路清楚

- 前端主入口集中在 `src/pages/PreflightStudioPage.tsx`，用户能从素材输入、手机预览、模拟评论、推荐回复和行动建议形成完整闭环。
- 后端入口 `server/routes/preflightSimulations.ts` 提供 `POST /api/preflight-simulations` 和 `GET /api/preflight-simulations/:jobId`，适合长耗时模型调用。
- `server/index.ts` 同时保留旧 sandbox 路由和新 preflight 路由，演示兼容性较好。

2. 工程骨架比普通 Demo 扎实

- `shared/preflightSimulation.ts` 把请求、结果、job 状态、V2 输出结构和远端结果归一化放在共享层，前后端契约比较稳。
- `server/lib/preflight/runPreflightSimulation.ts` 对真实 provider 失败有 degraded/fallback 语义，不会把失败伪装成正常成功。
- `harness/run-preflight-harness.ts` 和 fixture 让 preflight 主链路可以脱离真实 key 做稳定回归。

3. 测试覆盖已经有基本纵深

- `npm test` 当前 121 项通过，覆盖 schema、hooks、server store、orchestration、preflight provider、fixture 解析和页面静态渲染。
- `npm run verify:fixtures` 与 `npm run harness:preflight` 都能通过，说明 mock 回归链路可用。

4. 产品表达有记忆点

- “发布前试映场 / KOC Growth Lab / Campus AI Submission Studio”的表达比传统分析报告更容易被理解。
- 强制展示 `core / broad / weak / misfire` 四类人群是好设计，因为它承认平台推流不精确，避免把 AI 输出包装成确定预测。

## 主要问题

1. 安全依赖仍有阻断项

`npm audit --audit-level=moderate` 当前失败，返回 4 个漏洞：3 个 moderate、1 个 high。关键项包括：

- `path-to-regexp <0.1.13` high，来自 Express 4 依赖链，可通过 `npm audit fix` 尝试修复。
- `esbuild <=0.24.2` moderate，来自 Vite 依赖链，audit 建议 `--force` 到 Vite 8，属于破坏性升级，需要单独评估。
- `postcss <8.5.10` moderate，可通过常规 audit fix 尝试修复。

2. 还缺 CI 入口

项目根目录未发现 `.github` 工作流。当前本地验证命令比较完整，但没有自动化门禁，后续多人协作时容易再次出现 typecheck 或 build 回退。

建议最小 CI：

```bash
npm ci
npm run typecheck
npm test
npm run build
npm run verify:fixtures
```

3. 当前工作区改动面很大

`git status --short` 显示大量已修改文件和新增文件，包含主链路、旧 sandbox、examples、docs、harness、public、server/preflight、shared/preflight 等。当前可运行，但回滚和评审边界不够清楚。

建议把改动拆成至少三组：

- Preflight 主链路：`src/pages/PreflightStudioPage.tsx`、`src/hooks/usePreflightSimulation.ts`、`src/api/preflight.ts`、`server/lib/preflight/*`、`server/routes/preflightSimulations.ts`、`shared/preflightSimulation.ts`
- 验证与资产：`harness/*`、`public/preflight-fixtures/*`、`examples/*`
- 文档与交付记录：`README.md`、`docs/*`

4. 运行态仍偏 Demo

- `server/lib/preflight/preflightJobStore.ts` 使用内存 `Map` 和 30 分钟 TTL，适合本地演示，不适合多实例部署或服务重启后的任务恢复。
- `server/index.ts` 只设置了 `express.json({ limit: '12mb' })`，还没有鉴权、限流、请求来源约束和上传网关一致性配置。
- `.env.example` 默认真实 provider，但 key 为空。没有 key 时会走 degraded/fallback 语义，演示前必须明确 mock 还是真实 provider。

5. 旧链路边界需要继续收口

README 已说明旧 sandbox / impact scan 是兼容层，但代码层仍保留大量旧 orchestration、sandbox、variable impact scan 文件。短期没问题，长期需要明确“哪些是主产品，哪些是历史兼容”，否则维护成本会继续上升。

## 本地验证

- `npm run typecheck`：通过。
- `npm test`：通过，121/121。
- `npm run build`：通过，client/server 构建成功。
- `npm run verify:fixtures`：通过，输出 `Fixture verification passed.`。
- `npm run harness:preflight`：通过，输出 `Verified campus-coffee-demo`。
- `npm audit --audit-level=moderate`：失败，4 个漏洞。

说明：初次并行扫描时曾捕捉到 `PreflightStudioPage.tsx` 的 props 类型断点；当前工作区内该组件已接通 `insertedReplyIds` / `onInsertReply`，复跑 `npm run typecheck` 已通过。

## 风险清单

- P0：依赖漏洞未修复，`npm audit` 失败。
- P1：没有 CI，质量门禁依赖人工本地执行。
- P1：工作区改动过大，后续评审和回滚风险高。
- P1：真实 provider 依赖本地 `.env.local`，演示前需要固定 provider 策略。
- P2：内存 job store、无鉴权/限流、无持久化，不适合直接生产化。
- P2：旧 sandbox 与新 preflight 并存，产品边界需要继续收口。

## 建议优先级

1. 先修安全依赖：执行 `npm audit fix`，再跑 `typecheck/test/build/harness`。
2. 增加 CI：用最小 GitHub Actions 覆盖 `npm ci`、类型检查、单测、构建和 fixture 校验。
3. 拆分当前 dirty worktree：把主链路、验证资产、文档三类改动分开提交或至少分开评审。
4. 演示前固定环境策略：本地展示用 `PREFLIGHT_PROVIDER=mock`，真实演示用完整 Doubao/Ark key 和 endpoint id。
5. 若要上线，先补持久化 job store、限流、鉴权、日志和 provider 失败告警。
