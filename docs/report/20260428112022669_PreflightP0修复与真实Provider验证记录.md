# Preflight P0 修复与真实 Provider 验证记录

时间：2026-04-28 11:20:22

## 结果

已修复三类 P0 问题：模型 JSON 残片外露、校园提交场景 UI 混用增长话术、低分结果缺少强制补齐提示。最终本地验证通过，Browser Use 跑通宿舍与校园真实 provider 链路。

## 改动

- `shared/preflightSimulation.ts`：在共享字符串解析入口清理模型夹带的 `{"sentiment":...}`、`{"replyType":...}` 等对象残片，并保留“存疑”等可读语义。
- `src/pages/PreflightStudioPage.tsx`：按 KOC 增长 / 校园提交场景拆分标题、按钮、进度、评论区、右侧控制台、指标、动作列表和人群分层文案。
- `src/styles/pages.css`：新增低分质量检查样式，展示“必须补齐 / 必须复核”清单。
- `server/lib/preflight/prompt.ts`：quick 模式改为紧凑输出，降低真实 provider 超时概率；按场景要求 disclaimer。
- `server/lib/preflight/runPreflightSimulation.ts`：服务端归一化 `simulatedMetrics.disclaimer`，避免校园场景出现“播放、点赞、涨粉承诺”。
- `shared/preflightSimulation.test.ts`、`server/lib/preflight/runPreflightSimulation.test.ts`：补充 JSON 残片清理和校园 disclaimer 归一测试。

## 验证

- `npm run typecheck`：通过。
- `npm test`：121/121 通过。
- `npm run build`：通过。
- Browser Use 宿舍真实 provider：`provider=doubao`、`fallback=false`、评论区无 JSON 残片、无提交评审文案串台、无 console error。
- Browser Use 校园真实 provider：第一次完整结构修复后两次触发 150s timeout fallback；压紧 quick prompt 后复测成功，`provider=doubao`、`fallback=false`、有“必须补齐”、有“提交补齐动作”、无增长控制台/涨粉动作串台、无 JSON 残片、无 console error。

## 观察

- 修复后输出质量更像“提交前评审工具”，不再像社媒增长工具换皮。
- 校园真实 provider 仍偏慢，compact prompt 后可跑通，但耗时接近 100 秒；后续要继续做耗时监控、缓存和分阶段返回。
- 宿舍案例输出更自然，评论与推荐回复能直接用于内容调整；但个别指标如“关注转化率 100”仍需要做数值上限和口径约束。

## 风险

- quick 模式让远端先输出紧凑核心结果，服务端再补齐弱相关和误推覆盖；这能提升成功率，但部分补齐内容可能来自本地兜底，需要在 UI 上继续区分“远端原生 / 服务端补齐”。
- 当前 Browser Use 真实 provider 测试依赖 Ark/Doubao 当时响应状态，仍存在偶发超时。
