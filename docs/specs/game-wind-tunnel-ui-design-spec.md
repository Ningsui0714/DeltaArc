# 游戏产品风洞 UI 设计规格

## 1. 文档信息

- 文档版本: v0.1
- 更新时间: 2026-03-12
- 适用前端: React + Vite
- 运行方式: localhost Web UI
- 关联文档:
  - `product-wind-tunnel-mvp-prd.md`
  - `game-wind-tunnel-api-data-architecture.md`

## 2. 设计目标

这不是一个普通后台管理系统，也不是一个花哨的 AI 聊天壳。

它应该更像一个 `游戏策划作战台`：

- 有分析感
- 有实验室感
- 有策略推演感
- 不是企业 BI 风，也不是 ChatGPT 风

用户打开后应该立刻感受到：

- 这里是在判断一个策划点子能不能成立
- 这里重点是“玩家接受度”和“验证优先级”
- 这里是做决策，不是看流水账

## 3. 整体视觉方向

### 3.1 设计关键词

- tactical
- editorial
- simulation lab
- dark control room
- game design workstation

### 3.2 视觉气质

整体采用 `深色策略工作台` 风格，但不是传统炫酷霓虹黑客风。

目标视觉：

- 深墨色背景
- 冷灰蓝面板
- 暖橙或酸性黄作为风险和决策强调色
- 少量青绿色用于“可行/通过/正向信号”
- 信息排布像战术控制界面，而不是表单堆积

### 3.3 用户记忆点

最应该被记住的一点：

`它像一个把策划想法拆开验尸的战术桌。`

页面不应该平平地摆卡片，而应该有明显的“主战区”和“侧边情报区”。

## 4. 风格基线

### 4.1 色彩系统

建议用 CSS 变量统一。

```css
:root {
  --bg-app: #111417;
  --bg-elevated: #171c21;
  --bg-panel: #1d242b;
  --bg-panel-soft: #222b33;
  --line-subtle: rgba(255, 255, 255, 0.08);
  --line-strong: rgba(255, 255, 255, 0.18);
  --text-primary: #f3efe7;
  --text-secondary: #aeb7c2;
  --text-muted: #72808f;
  --accent-risk: #ff8f3f;
  --accent-alert: #ffd24a;
  --accent-good: #63d3a6;
  --accent-info: #6eb6ff;
  --accent-deep: #334250;
}
```

使用原则：

- 主背景保持稳定深色
- 高价值信息用亮文字
- 重点判断只用少量强调色，避免满屏彩色标签
- 风险用橙，不用红色泛滥

### 4.2 字体方向

避免默认系统字体和常见 AI 审美组合。

建议字体组合：

- 展示标题: `Bebas Neue` 或 `Oswald`
- 正文与数据: `IBM Plex Sans`
- 数字与标签: `IBM Plex Mono`

使用建议：

- 顶部大标题偏窄、偏硬朗
- 说明文字保持克制
- 关键数值、风险标签、模式标记用等宽字体增强“分析台”气质

### 4.3 圆角与阴影

- 主面板圆角: `18px`
- 小卡片圆角: `14px`
- 标签圆角: `999px`
- 阴影偏弱，更多依赖边框和层级对比

这套产品不适合大面积柔软投影，不然会偏 SaaS 模板味。

## 5. 信息架构

整体采用 `左侧项目导航 + 中央主工作区 + 右侧分析侧栏` 的结构。

```text
┌────────────┬──────────────────────────────┬───────────────┐
│ 左侧导航   │ 中央主工作区                 │ 右侧情报栏    │
│ Projects   │ 当前页面核心任务             │ 风险/证据/状态│
│ Stages     │ 表单 / 建模 / 对比 / 报告    │ 快速摘要      │
└────────────┴──────────────────────────────┴───────────────┘
```

### 5.1 左侧导航

固定宽度，承担两个任务：

- 项目层导航
- 当前工作流步骤导航

建议模块：

- Logo / 产品名
- 当前项目名称
- 模式标签: `Concept / Validation / Live`
- 页面步骤
  - 项目概览
  - 证据输入
  - 建模结果
  - 策略对比
  - 最终报告
- 历史项目入口

### 5.2 中央主工作区

这是最核心区域。

规则：

- 同一时刻只突出一个主要任务
- 表单和结果不要并列成平均两栏
- 页面上方总要有“当前结论”或“当前目标”

### 5.3 右侧情报栏

这是本产品区别于普通表单工具的关键。

建议固定展示：

- 当前证据等级
- 当前主要风险
- 当前玩家接受度提示
- 最近一次分析时间
- 当前最推荐下一步

右栏应该始终像一个战术摘要，而不是广告位。

## 6. 页面设计

## 6.1 项目概览页

### 目标

让用户快速进入一个策划问题。

### 页面结构

顶部：

- 项目标题
- 游戏类型
- 平台标签
- 模式标签
- 主操作按钮 `开始分析`

中部主区：

- 一句话概念
- 本次要验证的策划想法
- 核心循环摘要
- 当前判断状态

底部：

- 最近三次推演记录
- 最近生成报告列表

### 视觉重点

概览页不应该像文件详情页，而应该像一个“任务简报”。

## 6.2 证据输入页

### 目标

收集策划想法和支持材料。

### 页面布局

中央分成上下两段：

- 上半段: 结构化表单
- 下半段: 证据输入区

### 核心模块

- 基础信息卡
- 玩法描述卡
- 玩家画像卡
- 风险担忧卡
- 证据流列表

### 交互建议

- 输入区支持 `快速粘贴`
- 每条材料保存后立即变成一个 `证据卡`
- 证据卡有类型标签和可信度来源标签

### 设计重点

证据输入页要避免“巨大表单恐惧”，所以要做成一组连续卡片，而不是一页从上到下全是输入框。

## 6.3 建模结果页

### 目标

让用户理解系统当前是怎么“看待这个点子”的。

### 页面结构

第一屏：

- 玩法状态卡
- 证据等级
- 一句话系统判断

第二屏：

- 玩家角色卡矩阵
- 核心假设列表
- 缺口提醒

### 核心组件

#### 玩法状态卡

这是首页级组件，建议视觉上最大。

内容：

- 核心乐趣强度
- 上手理解难度
- 差异化程度
- 接受风险
- 原型验证成本

展示方式：

- 不做传统仪表盘
- 推荐做横向刻度条 + 简短解释

#### 玩家角色卡

每张卡展示：

- 角色名
- 动机
- 接受触发点
- 反感触发点
- 代表性判断

#### 核心假设列表

列表项要能一眼区分：

- 哪条假设最关键
- 哪条假设最缺证据
- 哪条假设最适合先验证

## 6.4 策略对比页

### 目标

帮助用户在 2 到 3 条策略中做决策。

### 页面结构

顶部：

- 页面标题
- 比较说明
- 主按钮 `运行推演`

主体：

- 三列策略卡对比区
- 下方结果对照区

### 组件要求

#### 策略卡

包含：

- 策略标题
- 类型标签
- 实施成本
- 见效周期
- 目标玩家
- 主要风险

#### 推演结果带

每条策略下方统一展示：

- 接受度
- 理解成本
- 留存潜力
- 执行风险
- 推荐结论

### 视觉重点

不要用普通表格。

应该做成 `并列战术卡`，让用户天然想比较。

## 6.5 最终报告页

### 目标

输出可以直接拿去内部讨论或自己决策的结果。

### 页面结构

第一屏：

- 最终结论
- 推荐动作
- 证据等级
- 当前置信度

第二屏：

- 玩家接受度分析
- 策略对比摘要
- 关键风险

第三屏：

- 未来两周验证计划
- 需要补的证据
- 报告导出按钮

### 视觉重点

报告页不是文档预览器。

要先做“结论优先”的可视页面，再提供 Markdown 导出。

## 7. 核心组件系统

## 7.1 AppShell

负责全局布局：

- 左导航
- 顶部状态栏
- 主工作区
- 右情报栏

## 7.2 SectionHeader

统一每个大区块的标题风格。

包含：

- 序号
- 标题
- 简短解释
- 可选操作按钮

## 7.3 SignalBadge

用于展示：

- `HIGH RISK`
- `LOW EVIDENCE`
- `WORTH PROTOTYPING`
- `NEEDS TEST`

风格建议：

- 等宽字体
- 全大写
- 半透明背景
- 左侧色条

## 7.4 StatRail

用于展示 0 到 1 的评估结果。

结构：

- 标签
- 数值
- 轨道
- 简短解释

适合替代传统环形图和雷达图。

## 7.5 EvidenceCard

用于证据输入与展示。

包含：

- 标题
- 来源类型
- 摘要
- 标签
- 删除/编辑操作

## 7.6 PersonaCard

用于展示玩家角色。

包含：

- 角色名
- 角色摘要
- 接受点
- 抗拒点
- 适用策略提示

## 7.7 StrategyCard

用于策略对比。

包含：

- 标题
- 类型
- 目标
- 成本
- 风险
- 结果摘要

## 7.8 SummaryPanel

放在右侧情报栏。

包含：

- 当前模式
- 当前证据等级
- 当前最关键风险
- 当前推荐下一步

## 8. 交互原则

### 8.1 减少跳转

用户应该在同一个工作台里完成大部分操作，尽量少进“详情页中的详情页”。

### 8.2 让结果可修正

系统生成的：

- 玩家角色
- 假设
- 策略
- 风险判断

都要允许用户手动编辑。

### 8.3 先摘要，后展开

每页第一屏都先给结论或判断摘要，再允许用户展开细节。

### 8.4 不做聊天优先

第一版不要把主界面做成聊天窗口。

聊天可以以后做，但主形态必须是工作台。

## 9. 动效建议

保持节制，但要有仪式感。

### 建议保留的动效

- 页面首次加载时主面板依次上浮
- 分析完成后右侧情报栏数字刷新
- 策略结果出现时做轻微滑入
- hover 时卡片边框亮度变化

### 不建议做的动效

- 复杂粒子背景
- 大面积持续脉冲
- 高频闪烁发光
- 过长的 loading 演出

## 10. 响应式策略

产品主场景还是桌面端。

### 桌面端

- 优先支持 `1280px` 以上
- 最佳宽度 `1440px - 1720px`

### 窄屏

- 左侧导航收缩为图标栏
- 右侧情报栏收进 drawer
- 三列策略卡改成纵向堆叠

### 手机端

第一版只保证能看，不追求完整编辑体验。

## 11. 页面状态设计

每个页面至少考虑这 5 类状态：

- empty
- editing
- loading
- generated
- error

### empty 状态

不要只写“暂无数据”。

应该明确提示：

- 现在缺什么
- 下一步该填什么
- 为什么这一步重要

### loading 状态

不要整页骨架屏泛滥。

建议展示：

- 当前正在做什么
- 已完成哪一步
- 还剩哪些步骤

更像一个分析流程状态，而不是网页在卡。

- React
- Vite

- React Hook Form
- Zod
- Tailwind CSS

## 12.3 UI 实现原则

- 页面组件只负责展示和交互
- 业务请求收敛到 `features` 或 `lib/api`
- 设计 token 单独维护
- 不要把样式碎片化到到处都是

## 13. 第一版最重要的 3 个页面

如果时间很紧，先把下面三个页面做对：

1. 证据输入页
2. 建模结果页
3. 策略对比页

原因：

- 它们最能体现差异化
- 它们最直接决定用户是否相信系统
- 它们最影响后续报告质量

## 14. 设计边界

这份设计需要避免以下问题：

- 太像企业后台
- 太像 AI 聊天产品
- 太像数据大屏
- 太像游戏启动器
- 太像常见浅色 SaaS 模板

正确方向应该是：

`策略工作台 + 游戏策划分析桌 + 本地实验室`

## 15. 最终设计结论

第一版 UI 最适合的关键词不是“酷炫”，而是：

- 硬朗
- 克制
- 有判断力
- 像一个真正拿来做决策的工具

如果后面开始正式开发，建议先做：

1. `AppShell`
2. `证据输入页`
3. `建模结果页`
4. `策略对比页`

这四部分一旦视觉和结构立住，整套产品的气质就定了。
