# 产品风洞材料上传说明

## 1. 当前版本支持什么

当前预览版已经支持 `文本粘贴 + 文件导入`，用于快速进入沙盒推演。

支持的文件类型：

- `.json`
- `.md`
- `.markdown`
- `.txt`

也就是说，这一版最适合上传这几类内容：

- 玩法草案
- 系统改动说明
- 玩家访谈摘录
- 竞品评论归纳
- 活动或商业化方案说明
- 指标快照的文字摘要

当前不建议直接上传的内容：

- 纯截图且没有文字说明
- 只有标题、没有上下文的零散摘录
- 超长原始聊天记录且没有筛选重点
- 大量重复材料

## 1.1 当前上传入口怎么选

### 项目概览页

适合上传：

- 项目包 JSON
- 设计文档 Markdown
- 产品需求说明 TXT

导入结果：

- 自动回填项目字段
- 如果文档里有材料内容，也会同步生成证据卡

### 证据输入页

适合上传：

- 玩家访谈摘录
- 竞品评论归纳
- 玩法草案
- 证据数组 JSON

导入结果：

- 只追加证据，不覆盖当前项目

## 2. 推荐的材料组织方式

建议一份材料只回答一类问题，不要把所有信息混成一大段。

推荐结构：

1. `材料标题`
2. `材料类型`
3. `来源`
4. `核心事实`
5. `你的判断或担忧`

推荐写法示例：

```md
标题：双人机关玩法草案
类型：design_doc
来源：策划内部草案

核心事实：
- 玩家在基地建设中会遇到需要两人同步完成的机关
- 成功后可以获得推进中期建造的关键资源
- 失败后当前版本会回退到上一个检查点

当前担忧：
- 单人补位方案还不明确
- 如果失败惩罚太重，可能会掩盖合作乐趣
```

## 3. 最好上传哪些“证据”

如果你想让沙盒推演更像 `mirofish` 那种“基于环境和角色做推演”，材料最好覆盖下面四层：

### A. 想法本身

- 你想验证的玩法/系统/活动是什么
- 你希望它改善什么
- 你认为它为什么会成立

### B. 玩家视角

- 目标玩家是谁
- 他们为什么会接受
- 他们为什么会反感
- 他们现在是怎么解决同类需求的

### C. 风险视角

- 学习成本
- 节奏风险
- 数值风险
- 题材和玩法不匹配风险

### D. 证据视角

- 玩家原话
- 竞品评论
- 留存/活跃/付费相关信号
- 团队资源和开发约束

## 4. 一次上传多少最合适

建议一次整理 `3 到 8 条高价值材料`，比一次扔几十条碎片更有效。

原因很简单：

- 太少，推演只会停留在概念判断
- 太多，噪音会盖过真正关键的假设

如果你只有很少信息，也可以先上传，但最好至少包括：

- 1 条玩法说明
- 1 条目标玩家说明
- 1 条主要担忧

## 5. 好材料和坏材料的区别

好材料：

- 有明确对象
- 有具体情境
- 有可推演的因果线索
- 能映射到“为什么成立 / 为什么失败”

坏材料：

- 太空泛
- 只有态度没有原因
- 只有结论没有过程
- 没有说明是哪个玩家群体

对比示例：

不推荐：

```text
玩家应该会喜欢合作玩法。
```

推荐：

```text
3 位轻协作玩家都表示愿意和朋友配合解谜，但前提是失败后能快速重开，不能重新跑 10 分钟流程。
```

## 6. 推荐命名方式

为了方便后续建模和报告归档，建议采用这个格式：

```text
[类型] 标题 - 来源 - 日期
```

例如：

```text
[design_doc] 双人机关玩法草案 - 内部策划稿 - 2026-03-12
[interview] 轻协作玩家反馈摘录 - Discord 访谈 - 2026-03-12
[review] 竞品差评归纳 - Steam 评论 - 2026-03-11
```

## 7. 当前预览版的最佳用法

当前预览版最推荐的使用方式是：

1. 先把文档中的关键内容整理成短文本
2. 逐条粘贴到“证据输入”页
3. 先跑一次 `快速推演`
4. 再用 `深度推演` 生成更完整的报告

如果后面接入正式文件上传，这份结构仍然适用，只是输入方式会从“手动粘贴”变成“文件解析后入库”。
