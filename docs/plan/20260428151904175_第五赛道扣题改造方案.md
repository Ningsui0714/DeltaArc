# 第五赛道扣题改造方案

## 目标

把当前项目从“发布前试映场 / 多场景预演工具”的表达，收敛成第五赛道作品本体：`KOC Growth Lab`。

赛道扣题句：

> 面向普通 KOC 的社媒 AI Agent。用户上传待发布内容后，AI 预演平台分发、评论区反应、涨粉机会和发布风险，并给出选题建议、发布策略、互动优化和账号成长动作。

## 非目标

- 不把产品改成通用 AI 文案生成器。
- 不承诺真实涨粉、播放量、点赞量。
- 不主讲校园比赛提交检查器。
- 不做刷量、诱导互动、违规引流。

## P0：提交前必须改

### 1. 统一产品定位

负责文件：

- `README.md`
- `README.en.md`
- `index.html`
- `package.json`
- `src/pages/PreflightStudioPage.tsx`

改造内容：

- 主标题统一为 `KOC Growth Lab`。
- 副标题统一为“第五赛道作品：面向普通 KOC 的社媒 AI Agent，发布前预演评论、涨粉机会和风险护栏”。
- README 第一段从“发布前试映场”改为“KOC 社媒增长预演 Agent”。
- `Campus AI Submission Studio` 只保留为隐藏/内部自检模式，不在默认首页、默认 demo、录屏主线中出现。

验收标准：

- 首屏 5 秒内能看懂：这是给普通 KOC 涨粉的 AI Agent。
- 页面默认状态不出现“校园提交检查 / 评审追问 / 提交补齐”作为主语。
- README 前 20 行明确出现第五赛道、KOC、社媒 AI Agent、涨粉。

### 2. 重做默认 Demo

负责文件：

- `src/pages/PreflightStudioPage.tsx`
- `server/lib/preflight/mockProvider.ts`
- `harness/fixtures/*`
- `public/*` 或示例图片目录

改造内容：

- 默认 demo 使用一个普通 KOC 场景，例如“宿舍桌面改造”“校园平价穿搭”“期末复习效率工具”。
- 输入内容必须包含：
  - 平台：小红书或视频号。
  - 账号定位：普通学生 KOC。
  - 目标：涨粉、收藏、评论互动。
  - 待发布图文/视频脚本。
  - 账号边界：不能夸大效果、不能诱导互动、素材需原创。
- 删除默认演示里的比赛提交三件套、评审追问、官方结论等语义。

验收标准：

- 点击“一键载入示例”后，表单内容就是第五赛道 KOC 增长案例。
- mock provider 输出稳定包含：内容方向、选题建议、发布策略、互动优化、账号成长动作。
- harness fixture 可以稳定回归同一条第五赛道 demo。

### 3. 结果区按赛题四个关键词重排

负责文件：

- `src/pages/PreflightStudioPage.tsx`
- `src/styles/pages.css`
- `shared/preflightSimulation.ts`

改造内容：

把结果区从“发布前报告”重排成五个扣题模块：

1. 内容方向：AI 读图/读稿，判断这条内容的核心卖点、第一眼吸引点、误读点。
2. 选题建议：给出下一条内容、系列化方向、A/B 选题实验。
3. 发布策略：标题、封面、发布时间、平台标签、正文结构。
4. 互动优化：置顶评论、回复话术、评论区分歧处理、潜在新粉转化点。
5. 账号成长：人群分层、关注理由、48 小时后是否追更、复盘指标。

安全审查模块改名：

- 从“发布安全审查”改为“发布风险护栏”。
- 文案强调“帮 KOC 避免翻车”，不强调“大厂法务审查”。

验收标准：

- 页面结果标题直接覆盖赛题要求里的“内容方向、选题建议、发布策略、互动优化”。
- “下一条内容/系列选题”不藏在动作列表里，而是独立可见。
- 安全审查不抢主功能，只作为增长前的风险护栏。

### 4. Prompt 与 mock 强制扣题

负责文件：

- `server/lib/preflight/prompt.ts`
- `server/lib/preflight/mockProvider.ts`
- `shared/preflightSimulation.ts`
- `shared/preflightSimulation.test.ts`
- `server/lib/preflight/runPreflightSimulation.test.ts`

改造内容：

- prompt 明确模型角色：`KOC Growth Lab, social growth simulation agent for ordinary KOC creators`。
- 输出 contract 明确必须返回：
  - contentDirection
  - topicIdeas
  - publishStrategy
  - interactionOptimization
  - accountGrowthPlan
  - publishRiskGuardrail
- mock provider 对 `follower_growth` 必须返回第五赛道结构。
- 测试断言不得只检查“涨粉动作”，要检查四个赛题关键词。

验收标准：

- `npm test` 能证明第五赛道结构没有丢。
- 真实 provider 即使发挥，也会被 schema 拉回 KOC 增长语义。
- 校园提交分支不会污染 `follower_growth` 输出。

### 7. 准备提交物料

负责文件：

改造内容：

PDF 结构建议：

## 实施顺序

1. P0-1：先统一定位和默认入口，避免第一眼跑题。
2. P0-2：替换默认 demo 和 mock 输出，让演示样例完全贴赛题。
3. P0-3：重排结果区，把赛题关键词变成页面模块。
4. P0-4：更新 prompt/schema/tests，避免真实 provider 输出漂移。
5. P1-5/P1-6：增强账号成长计划和风险护栏。
6. P1-7：制作 PDF 与录屏脚本。

- `npm run typecheck`
- `npm test`

- `npm run verify:fixtures`
- `npm run harness:preflight`

- 提交物料检查：在线 Demo、PDF、3 分钟内横屏 MP4，三项齐全。

## 风险与回滚

- 风险：过度强调风险护栏会偏离“涨粉”。回滚方式：安全模块降级为结果区最后一张卡，不放首屏。
- 风险：新增 schema 字段影响真实 provider 成功率。回滚方式：先在 parser 层做 legacy fallback，再逐步收紧 prompt。
- 风险：隐藏校园提交模式影响已有自检能力。回滚方式：保留 query 参数或开发入口，不在默认提交链路露出。

## 决策

按第五赛道提交时，项目名使用 `KOC Growth Lab`，主功能顺序为：

1. 内容方向
2. 选题建议
3. 发布策略
4. 互动优化
5. 账号成长
6. 发布风险护栏

`发布前试映场` 是底层能力描述，不作为参赛主标题。
