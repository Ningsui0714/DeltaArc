# Preflight 输出结构 V2 设计方案

时间：2026-04-27 19:44:49

## 结果

本方案将当前 Preflight 输出从“粗粒度模拟报告”升级为“可审阅、可执行、可解释的发布前预演结果”。核心目标是解决三类问题：

- 模型输出质量难判断：不知道哪些结论可靠，哪些只是推测。
- 评论、回复、动作混在一起：导致前端曾出现校园 Demo 串宿舍模板。
- 建议不够可执行：用户知道“要优化封面”，但不知道具体改成什么。

建议采用 `PreflightSimulationResultV2`，先兼容 V1 读写，再逐步切 UI。

## 目标

- 每个输出字段都能对应一个明确 UI 区块。
- 每条评论都拆出“表层评论、真实意图、推荐回复、内容改法”。
- 每条修改建议都给出具体改写样例。
- 明确模拟值、可信信号、弱信号和需要人工复核的点。
- 同时支持 KOC 涨粉和校园 AI 提交前试映，不让场景话术串台。

## 非目标

- 不承诺真实流量、真实涨粉数或官方评审结果。
- 不把模型输出当成最终发布内容，仍要求用户审核。
- 不在本阶段引入复杂历史学习或账号私域数据。

## 当前 V1 问题

当前结构：

```ts
PreflightSimulationResult {
  contentRead;
  imageInsight;
  pushModel;
  simulatedReplies;
  risks;
  interventions;
  confidence;
}
```

主要问题：

- `simulatedReplies.intervention` 同时承担“怎么回评论”和“怎么改内容”，职责混乱。
- `interventions.action` 没有 `exampleRewrite`，用户不能直接照着改。
- `confidence` 太粗，只给整体置信度，不说明哪些信号可靠。
- `pushModel` 更像分层摘要，缺少每类人为什么被推、为什么流失。
- `imageInsight` 有读图结果，但没有拆出“首屏强信号/误读点/缺少标注”。
- 页面互动数据缺少“模拟值”语义，容易被误解成预测。

## V2 总体结构

```ts
type PreflightSimulationResultV2 = {
  schemaVersion: 'preflight_v2';
  generatedAt: string;
  provider: 'doubao' | 'mock';
  model: string;
  mode: PreflightSimulationMode;
  scenario: PreflightScenario;
  degraded: boolean;
  fallbackReason?: string;

  contentPromise: ContentPromise;
  visualRead: VisualRead;
  audienceDistribution: AudienceCohortV2[];
  simulatedReplies: SimulatedReplyV2[];
  interventions: InterventionV2[];
  riskReview: RiskReview;
  simulatedMetrics: SimulatedMetrics;
  qualityCheck: QualityCheck;
  warnings: string[];
};
```

## 场景识别

```ts
type PreflightScenario =
  | 'koc_growth'
  | 'campus_submission'
  | 'brand_post'
  | 'generic_preflight';
```

规则：

- `goal=follower_growth` -> `koc_growth`
- `platform=campus_ai_competition` 或 `goal=submission_readiness` -> `campus_submission`
- 其他营销/转化目标 -> `brand_post`
- 兜底 -> `generic_preflight`

用途：

- 决定 UI 文案、prompt 指令和推荐回复语气。
- 防止校园 Demo 使用宿舍/涨粉模板。

## 1. 内容承诺层

```ts
type ContentPromise = {
  oneLinePromise: string;
  targetUser: string;
  userPain: string;
  promisedValue: string;
  desiredAction: string;
  proofProvided: string[];
  proofMissing: string[];
  overclaimRisk: string;
};
```

用途：

- 回答“这条内容到底在承诺什么”。
- 帮用户检查标题/正文/图片是否一致。

示例：

```json
{
  "oneLinePromise": "99元把宿舍桌面整理成可复制的学习角",
  "targetUser": "宿舍学生和备考人群",
  "userPain": "桌面乱、预算少、想知道具体怎么买",
  "promisedValue": "低预算清单和真实踩坑",
  "desiredAction": "收藏清单、评论痛点、关注下一期",
  "proofProvided": ["前后对比图", "99元标题"],
  "proofMissing": ["新增单品价格明细", "尺寸信息"],
  "overclaimRisk": "用户可能误解为整套桌子只花99元"
}
```

## 2. 视觉读片层

```ts
type VisualRead = {
  firstGlance: string;
  visibleText: string[];
  visibleObjects: string[];
  strongestSignal: string;
  confusionPoints: string[];
  missingAnnotations: string[];
  coverScore: number;
  mobileLegibility: 'good' | 'medium' | 'poor';
};
```

用途：

- 把“AI 看到了什么”从泛泛描述变成首屏诊断。
- 支持 UI 展示封面缺什么标注。

验收：

- 必须包含至少 1 个 `strongestSignal`。
- 如果图上有文字，`visibleText` 必须尽量提取。
- `confusionPoints` 不能只写“信息不清晰”，必须说明哪里不清晰。

## 3. 分发人群层

```ts
type AudienceCohortV2 = {
  id: string;
  tier: 'core' | 'broad' | 'weak' | 'misfire';
  userProfile: string;
  exposureShare: number;
  pushReason: string;
  likelyReaction: string;
  conversionTrigger: string;
  dropOffReason: string;
  misunderstandingRisk: string;
};
```

变化：

- V1 的 `whyPushed / likelyBehavior / conversionIntent` 保留语义，但命名更直接。
- 新增 `dropOffReason`，让用户知道为什么划走。

UI 映射：

- 手机预览下方：展示 `userProfile + likelyReaction`。
- 右侧控制台：展示 `conversionTrigger / dropOffReason`。

## 4. 评论模拟层

```ts
type SimulatedReplyV2 = {
  id: string;
  cohortId: string;
  userType: string;
  tier: 'core' | 'broad' | 'weak' | 'misfire';
  sentiment: 'positive' | 'neutral' | 'skeptical' | 'negative' | 'irrelevant';
  replyType:
    | 'question'
    | 'objection'
    | 'save_signal'
    | 'follow_signal'
    | 'share_signal'
    | 'misread'
    | 'scroll_away';
  comment: string;
  surfaceIntent: string;
  hiddenNeed: string;
  conversionSignal: string;
  riskSignal: string;
  suggestedReply: string;
  contentAction: string;
  evidenceNeeded: string[];
};
```

关键变化：

- `comment`：用户看到的模拟评论。
- `surfaceIntent`：这句话表面在问什么。
- `hiddenNeed`：背后的真实需求。
- `suggestedReply`：可直接复制的评论区回复。
- `contentAction`：这条评论暴露出的内容修改动作。
- `evidenceNeeded`：需要补什么证据。

示例：

```json
{
  "userType": "准备改造的新生",
  "comment": "99元是只买这些小东西还是连桌子一起啊？",
  "surfaceIntent": "确认预算口径",
  "hiddenNeed": "担心标题党，想知道清单是否真实可复制",
  "conversionSignal": "如果解释清楚，可能收藏并关注清单",
  "riskSignal": "预算误解会引发负面评论",
  "suggestedReply": "99元只算本次新增的收纳和灯具，桌子和电脑是原有物品。我会把新增单品价格表放到下一篇。",
  "contentAction": "正文第一段补一句“99元只包含新增收纳/灯具，不含原有桌椅电脑”。",
  "evidenceNeeded": ["新增单品价格表", "原有物品说明"]
}
```

验收：

- `suggestedReply` 必须像真人回复，不允许只写“立即回复告知使用方式”。
- `contentAction` 必须说清楚改哪里。
- `comment` 不能重复模板句，每条评论必须有不同意图。

## 5. 修改动作层

```ts
type InterventionV2 = {
  id: string;
  priority: 'P0' | 'P1' | 'P2';
  target:
    | 'cover'
    | 'title'
    | 'opening'
    | 'body'
    | 'comment'
    | 'proof'
    | 'next_post'
    | 'timing';
  problem: string;
  change: string;
  exampleRewrite: string;
  expectedEffect: string;
  effort: 'low' | 'medium' | 'high';
  evidenceSource: string;
};
```

关键变化：

- 用 `problem/change/exampleRewrite` 替代 V1 的 `action/reason/expectedChange`。
- `exampleRewrite` 是强制字段，必须给用户可直接拿去改的样例。
- `evidenceSource` 指向来自哪条评论、读图风险或内容承诺缺口。

示例：

```json
{
  "priority": "P0",
  "target": "title",
  "problem": "用户可能误解99元包含整套桌椅",
  "change": "标题明确新增单品口径",
  "exampleRewrite": "99元新增单品清单：宿舍桌面从乱到可学习",
  "expectedEffect": "减少标题党质疑，提高收藏和关注清单的转化",
  "effort": "low",
  "evidenceSource": "预算误解评论 + visualRead.missingAnnotations"
}
```

验收：

- P0 至少 2 条。
- 每条必须有 `exampleRewrite`。
- `problem` 和 `expectedEffect` 不能重复。

## 6. 风险复核层

```ts
type RiskReview = {
  topRisks: RiskItemV2[];
  complianceRisks: RiskItemV2[];
  misreadRisks: RiskItemV2[];
};

type RiskItemV2 = {
  title: string;
  severity: 'low' | 'medium' | 'high';
  trigger: string;
  likelyUserComment: string;
  mitigation: string;
};
```

用途：

- 把传播风险、合规风险、误读风险分开。
- 校园 Demo 场景必须把“非官方工具”列入合规风险。

## 7. 模拟指标层

```ts
type SimulatedMetrics = {
  disclaimer: string;
  attentionScore: number;
  saveIntentScore: number;
  commentIntentScore: number;
  followIntentScore: number;
  shareIntentScore: number;
  metricCards: Array<{
    label: string;
    simulatedValue: number;
    rationale: string;
  }>;
};
```

变化：

- 不再只展示点赞/收藏/评论/转发数字。
- 每个数字必须带 `rationale`，并统一标记为模拟值。

UI 文案：

```text
以下为发布前模拟值，不代表真实播放、点赞或涨粉承诺。
```

## 8. 质量检查层

```ts
type QualityCheck = {
  overallScore: number;
  reliableSignals: string[];
  weakSignals: string[];
  possibleHallucinations: string[];
  needsHumanReview: string[];
  outputWarnings: string[];
};
```

用途：

- 让用户知道“这份结果哪里可信，哪里要人工看”。
- 支持前端展示质量徽章。

示例：

```json
{
  "overallScore": 74,
  "reliableSignals": ["图中文字清晰", "评论围绕预算和清单"],
  "weakSignals": ["没有真实账号历史数据", "互动数字仅为模拟"],
  "possibleHallucinations": ["模型可能误判单品是否为新增"],
  "needsHumanReview": ["价格口径", "是否可以使用大赛/官方相关表述"],
  "outputWarnings": ["不要把模拟值当成真实承诺"]
}
```

## V1 到 V2 迁移

### 后端解析

新增：

```ts
parsePreflightSimulationResultV2(value): PreflightSimulationResultV2
```

兼容策略：

- Provider 返回 V2：直接解析。
- Provider 返回 V1：通过 adapter 转为 V2。
- 旧 UI 暂时继续读 V1 字段，新 UI 逐步切到 V2。

### Adapter 规则

```ts
contentRead -> contentPromise
imageInsight -> visualRead
pushModel.cohorts -> audienceDistribution
simulatedReplies.text -> simulatedReplies.comment
simulatedReplies.why -> simulatedReplies.hiddenNeed
simulatedReplies.intervention -> simulatedReplies.contentAction
interventions -> interventions
confidence -> qualityCheck
```

缺失字段用 conservative fallback，不允许编造敏感事实。

## Prompt 契约调整

必须加入的约束：

```text
Return strict JSON using schemaVersion="preflight_v2".
Do not use generic filler such as "看起来有点意思，但还需要更多信息".
Every simulated reply must have a different user intent.
suggestedReply must be a copy-ready comment reply, not an action instruction.
contentAction must say exactly which part to change.
Every intervention must include problem, change, exampleRewrite, expectedEffect, effort.
Numbers are simulated diagnostics, not predictions.
If a scenario is campus_submission, do not use dorm, shopping-list, makeover, or KOC-specific wording unless present in the input.
```

## 前端 UI 映射

### 手机预览

- 标题/正文：来自用户输入。
- 模拟评论区：`simulatedReplies.comment`。
- 点击评论后：
  - 表层意图：`surfaceIntent`
  - 隐藏需求：`hiddenNeed`
  - 推荐回复：`suggestedReply`
  - 内容动作：`contentAction`

### 右侧控制台

- 内容承诺：`contentPromise`
- 读片诊断：`visualRead`
- 修改动作：`interventions`
- 质量检查：`qualityCheck`
- 人群分层：`audienceDistribution`
- 模拟指标：`simulatedMetrics`

### 场景文案

`koc_growth`：

- “涨粉动作”
- “评论区回复”
- “下一条内容实验”

`campus_submission`：

- “提交前补洞”
- “评审追问”
- “材料完整性”
- “身份/合规边界”

## 实施计划

### P0：结构与串台修复

- 新增 V2 TypeScript 类型。
- 新增 V2 parser 和 V1 adapter。
- Prompt 切 V2。
- 前端推荐回复改读 `suggestedReply`。
- 校园 Demo UI 文案按 `scenario` 切换。

验收：

- 宿舍案例无校园话术。
- 校园案例无宿舍清单/桌面痛点串台。
- 每条评论都有 `suggestedReply` 和 `contentAction`。

### P1：动作质量提升

- `interventions` 强制 `exampleRewrite`。
- UI 展示改写前/改写后。
- 增加“模拟值”标记。

验收：

- 每条 P0/P1 建议用户能直接复制或执行。
- 不再出现“优化封面”“补充信息”这类空泛动作。

### P2：质量自检

- 展示 `qualityCheck`。
- Browser Use 回归加入质量断言：
  - 无泛化句。
  - 无场景串台词。
  - 有至少 2 条可复制改写。
  - 有模拟值 disclaimer。

## 验收标准

- `npm run typecheck` 通过。
- `npm test` 通过。
- `npm run build` 通过。
- Browser Use 跑通：
  - `?preflightCase=imagegen-dorm`
  - `?preflightCase=imagegen-campus-demo`
- 两个案例都满足：
  - `provider=doubao`
  - `status=completed`
  - 无 fallback/mock
  - 无场景串台
  - 有 `suggestedReply`
  - 有 `exampleRewrite`
  - 有 `qualityCheck`
  - 有模拟值 disclaimer

## 推荐落地顺序

先做 P0。因为 P0 能立刻解决最影响观感的问题：结构不清、推荐回复串台、回复和动作混在一起。

P1 再提高可执行性。P2 最后做质量仪表盘和自动化断言。
